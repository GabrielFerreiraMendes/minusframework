# Preparação para Release v0.1.0 — Revisão e Melhorias do MinusFrameWork-Meta

## Objetivo

Preparar o meta-repositório MinusFrameWork-Meta para a primeira release pública (v0.1.0) através de 3 fases sequenciais: Patch Rápido (desbloqueio), Fundação (automação/padronização) e Release-Quality (polimento final).

## Contexto

O MinusFrameWork-Meta é um meta-repo que agrega 9 submódulos (Core, ORM, Migrator, Messaging, FeatureFlags, Telemetry, Extensions, AI, CLI) do framework Delphi MinusFrameWork. Inclui site de documentação Docusaurus, license server Node.js, instalador Inno Setup e scripts de CI/CD.

## Fase 1 — Patch Rápido

### 1.1 Segurança — Chave Privada

**Problema:** `license-server/private.pem` existia no repositório.

**Solução executada:**
- Verificado que `private.pem` e `public.pem` nunca foram commitados (já ignorados por `license-server/*` no `.gitignore`)
- Adicionado `*.pem` global ao `.gitignore` como segurança adicional
- Regenerado par de chaves RSA 2048 via `node generate-keys.js`

### 1.2 Bloqueios de Compilação

**Investigação concluída:**
- `MF.FeatureFlags.Licensing.pas` — existe, sintaticamente correto (21 linhas, adapter simples)
- Identificados 5 gargalos potenciais:

| # | Problema | Impacto |
|---|----------|---------|
| a | Dependência circular Core ↔ FeatureFlags | Médio |
| b | Cross-submodule paths nos .dpr/.dpk dependem da estrutura correta | Alto |
| c | Ordem de compilação: Telemetry → Messaging → Core → FeatureFlags | Médio |
| d | Missing `{$R *.res}` em `MinusTelemetry_Runtime.dpk` e `MinusMessaging_Design.dpk` | Baixo |
| e | Search paths ausentes nos `.dproj` para submódulos vizinhos | Alto |

**Recomendações:**
- Quebrar dependência circular: mover `MF.FeatureFlags.Licensing.pas` para o Core ou usar Injeção de Dependência
- Documentar ordem de build e configurar search paths nos `.dproj`

### 1.3 Documentação — Correções

**Realizado:**
- Removida linha duplicada em `docs/getting-started.md` (linha 135)
- Substituído `[Campo` → `[Coluna` em 31 ocorrências em 8 arquivos
- Removido parâmetro de tamanho onde presente (ex: `[Campo('NOME', 100)]` → `[Coluna('NOME')]`)
- Nenhum conteúdo raso encontrado nos docs do ORM
- Nenhum link quebrado detectado

### 1.4 Configurações

**Realizado:**
- Migrado `onBrokenMarkdownLinks` para `markdown.hooks.onBrokenMarkdownLinks` (Docusaurus v4)
- Removida linha duplicada `docs-mkdocs/` do `.gitignore`
- Build do Docusaurus passa sem warnings ✅

## Fase 2 — Fundação

### 2.1 Automação de Release

Criar script `release.ps1` na raiz que:
1. Verifica saúde dos submódulos (`.git` presente, branch correta)
2. Executa build na ordem correta (Telemetry → Messaging → Core → FeatureFlags → Extensions → CLI/Migrator/ORM/AI)
3. Executa testes onde disponíveis (`run-tests.ps1` em cada submódulo)
4. Gera artefatos compilados em diretório `dist/`
5. Cria release notes a partir de `RELEASE_NOTES.md` + changelogs dos submódulos
6. Executa `build-installer.ps1` para gerar o instalador final

### 2.2 Ordem de Build Documentada

```
Nível 1: Telemetry (base)
Nível 2: Messaging (depende de Telemetry)
Nível 3: Core Runtime (depende de Messaging + Telemetry)
Nível 4: FeatureFlags, Extensions (dependem do Core)
Nível 5: CLI, Migrator, ORM, AI (dependem do Core)
```

Cada nível só compila se o anterior passou.

### 2.3 Padronização entre Submódulos

Alinhar os 9 submódulos em:
- `.gitignore` — regras mínimas consistentes (`*.dcu`, `*.exe`, `__history/`, `.local`)
- Scripts de teste — `run-tests.ps1` em cada submódulo seguindo mesmo padrão
- Estrutura de diretórios — `Source/`, `Tests/`, `Docs/`, `Packages/`

### 2.4 CI/CD — Pipeline GitHub Actions

Workflow `.github/workflows/release-prep.yml`:
- **trigger:** push para `main`, PRs para `main`
- **jobs:**
  - `docs`: build Docusaurus, deploy para GitHub Pages
  - `submodules`: verifica se todos os submódulos estão no commit esperado
  - `validate`: checa `.gitignore`, estrutura de diretórios, scripts de teste

## Fase 3 — Release-Quality

### 3.1 Documentação — Revisão Completa

- Revisar todos os 30+ arquivos em `docs/` para consistência terminológica
  - Verificar uso consistente de `[Tabela]`, `[Coluna]`, `[ChavePrimaria]` vs variações
  - Verificar se exemplos de código compilam com a API real
- Adicionar exemplos para módulos com documentação mais enxuta:
  - Messaging (consumidores, filas, exchanges)
  - FeatureFlags (providers, avaliação)
  - Telemetry (tracing, spans)
  - AI (MCP server, ferramentas)
- Verificar que todas as páginas do navbar/footer existem e têm conteúdo
- Garantir que `docs/index.md` lista todos os módulos corretamente

### 3.2 Instalador — Validação

- Executar `Installer\build-installer.ps1` do zero
- Verificar:
  - BPLs runtime e design-time são copiados para `C:\MinusFramework\Bpl`
  - DCPs para `C:\MinusFramework\Dcp`
  - DLLs (ORM, Migrator) para `C:\MinusFramework\Dll`
  - CLIs (`MinusMigrator.exe`, `minus.exe`) para `C:\MinusFramework\Cli`
  - Samples para `C:\MinusFramework\Samples`
- Validar que o Inno Setup script (`MinusFramework.iss`) não tem paths hardcoded quebrados

### 3.3 CLI — Readiness

- Verificar `Cli\Source\minus.dpr` — estrutura, comandos implementados
- Confirmar comandos do roadmap: `make:entity`, `new:api`, `generate:migration`
- Verificar doc em `docs/cli/` cobre todos os comandos existentes

### 3.4 Segurança — Auditoria Final

- Escanear repo por possíveis secrets:
  ```powershell
  Select-String -Path "*.pas","*.ps1","*.js","*.json","*.ts","*.yml","*.yaml" -Pattern "(password|secret|token|api.key|senha)" -Recurse
  ```
- Confirmar que `*.pem` está no `.gitignore` e não tracked
- Verificar scripts PowerShell para exposição de credenciais em parâmetros ou URLs
